Command Line Interface

  • Generating C bindings from the command line is simple.

  • Running Henka from the CLI:

    $ henka point.h

  • On this header:

    struct Point {
    float x;
    float y;
};

float point_magnitude(struct Point p);

Henka CLI

  • Generates point.nim:

    type
  struct_Point* {.bycopy, importc:"struct Point", header:"point.h".} = object
    x* :cfloat
    y* :cfloat
  Point* = struct_Point
proc point_magnitude*(p :struct_Point) :cfloat {.importc:"point_magnitude", cdecl, header:"point.h".}

Henka CLI

Generating bindings:

  • For C++:

    $ henka --cpp --std=c++17 header.hpp

  • For JS/TS:

    $ henka --js lib.js

Henka as a library

  • However, Henka is best used as a library.

import henka

let bindings = generate("header.h")
"bindings.nim".writeFile(bindings)

Customization is a first class citizen

  • Importing Henka as a library allows for complete customization of the output

proc rename(kind: henka.LabelKind, name: string): string =
  result = name.replace("point_", "")

proc pragmas(kind: henka.LabelKind, name: string, defaults: seq[(string, string)]): seq[(string, string)] =
  result = defaults

  if kind is henka.StructType:
    result.add [("pure", ""), ("inheritable", "")]

let bindings = generate(
  headerPath,
  renamer = rename,
  pragmaOverride = pragmas
)

Henka as a library

  • Generating a cleaner point.nim:

    type Point *{.bycopy, importc:"struct Point", header:"header.h", pure, inheritable.}= object
  x *:cfloat
  y *:cfloat
proc magnitude *(p :Point) :cfloat {.importc:"point_magnitude", cdecl, header:"header.h".}

Henka v1

Architecture

  • v1 generated Nim bindings for C using

    clang -ast-dump=json.

  • Initially, we wanted to keep Henka independent from libclang and its API.

  • Reason 1: easy to parse JSON data.

  • Reason 2: fewer binary dependencies, easier to port and get working anywhere.

Henka v1

How it worked

  • 1 - Walk the AST JSON data.

  • 2 - Handle edge cases: unnamed unions/structs, forward declarations, and enum value duplications.

  • 3 - Generate Nim code for types, variables, and functions.

  • Basic symbol renamer for customizing the output.

Henka v2

Starting over

  • We went back to using libclang.

  • Reason 1: easy to parse JSON data. libclang is clear enough, and supports extended features.

  • Reason 2: fewer dependencies... True, but bindings are generated once, and distributed via source control.

  • This is an design distinction from Futhark, which builds bindings in compile-time instead.

Henka v2

New features, more scale

  • v2 supports C, C++, and JS/TS.

  • Callbacks for pragmas and symbols: rename, override, filter, resolve unnamed, etc.

  • Supports generating bindings for both static and dynamically linked libraries.

  • Support for generating enums as cint, distinct cint, and pure/impure enums.

Henka v2

New features, more scale

  • C++ blew the scale out of proportion:

    • v1 edge cases
    , extern C
    , operator overloads
    , move semantics
    , type aliases
    , double and trailing _
    , duplicated typedefs
    , templates
    , static and instance methods
    , etc.

Henka v2

How it works

  • 1 - Uses libclang's visitor to walk the AST.

  • 2 - Handle edge cases, mapping C++ AST nodes to Nim AST nodes using astTF.

  • 3 - Pass the Nim AST to a nim code generator.

  • This allows Henka to focus exclusively on mapping ASTs, and nothing else.

Henka v2

Workflow

  • The same principles still apply, but with a much bigger scale (ie: C, C++, and JS/TS).

  • Test Driven Development is maximum priority. Unit Tests are Henka's specification.

  • Tests check for bugs, validate new features, and keep AI in check.

About AI usage:

  • We decided when to use AI on the basis of what's faster.

  • A lot of code was generated. But all code was left thorougly tested or strictly reviewed.

  • As it scales, the project becomes a token eating machine.
