Henka CLI

  • Generating C bindings from the command line is simple.

  • Running henka from the CLI:

    $ henka point.h

  • On this header:

    struct Point {
    float x;
    float y;
};

float point_magnitude(struct Point p);

Henka CLI

  • Generates point.nim:

    type
  struct_Point* {.bycopy, importc:"struct Point", header:"point.h".} = object
    x* :cint
    y* :cint
  Point* = struct_Point
proc point_magnitude*(p :struct_Point) :cfloat {.importc:"magnitude", cdecl, header:"point.h".}

Generating C++ and JS/TS bindings:

$ henka --cpp --std=c++17 --js header.hpp
$ henka --js lib.js

Henka as a library

  • Henka can be imported and called from Nim code.

import henka

let bindings = generate("header.h")
# write to a file

Refining the bindings

  • Calling it from Nim allows customizing how bindings are generated.

proc rename(kind: henka.LabelKind, name: string): string =
  result = name.replace("point_", "")

proc pragmas(kind: henka.LabelKind, name: string, defaults: seq[(string, string)]): seq[(string, string)] =
  result = defaults

  if kind is henka.StructType:
    result.add [("pure", ""), ("inheritable", "")]

let bindings = generate(
  headerPath,
  renamer = rename,
  pragmaOverride = pragmas
)

Henka v1

Architecture

  • v1 generated Nim bindings for C using

    clang -ast-dump=json.

  • Initially, we wanted to keep henka independent from libclang and its API'.

  • Reason 1: easy to parse JSON data.

  • Reason 2: less binary dependencies, easier to port and get working anywhere.

Henka v1

How it worked

  • 1 - Walk the AST json data.

  • 2 - Handle some edge cases such as unnamed unions/structs, forward declarations, and enum value duplications.

  • 3 - Generate Nim code for types, variables, and functions.

  • A basic symbol renamer was provided for customizing the output.

Henka v2

Starting over

  • We went back to using libclang.

  • Reason 1: easy to parse JSON data. As simple as parsing JSON is, clang's API provides clear enough semantics, and a visitor. Overall less parsing, this was crucial for v2's features, specially C++.

  • Reason 2: less binary dependencies... While still true, this is a 'do once' thing.

Henka v2

New features, more scale

  • v2 supports C, C++, and JS/TS.

  • Callbacks for working with symbols and pragmas (ie: renaming, overriding, filtering, unnamed symbols).

  • Can be configured to generate bindings for static or dynamic libraries.

  • Allows more customization on how enums are generated.

Henka v2

New features, more scale

  • C++ blew the scale out of proportion:

    • v1 edge cases
    , extern C
    , operator overloads
    , move semantics
    , type aliases
    , double and trailing _
    , duplicated typedefs
    , templates
    , static and instance methods
    , etc.

Henka v2

How it works

  • 1 - Uses libclang's visitor to walk the AST.

  • 2 - Handle edge cases, mapping C++ AST nodes to Nim AST nodes using astTF.

  • 3 - Pass the Nim AST to a nim code generator.

  • This allows henka to only focus on mapping ASTs and nothing else.

Henka v2

Workflow

  • We added TDD to the workflow, tests are now our specification.

  • Tests check for bugs, validate new features, and keep AI in check.
